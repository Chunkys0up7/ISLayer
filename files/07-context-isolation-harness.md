# Component 07 — Context Isolation Harness

## Purpose

**This is the highest-leverage component in the simulator.**

For any (u, v) handoff, execute v twice: once with the full rich context (as in normal grounded mode), and once with *only the formally declared outputs* from u. Compare outcomes. If they differ, the handoff is being carried by content or state outside the declared contracts — meaning the architecture is load-bearing on something undocumented, and will silently break when triples evolve independently.

This is the direct test of the core bet: that your CIM→PIM→PSM contracts are what compose, not the content that happens to be present.

## Inputs

- A pair `(u, v)` or a full journey trace.
- The seed context that produced the full-content outcome.
- LLM configuration.

## Outputs

- `IsolationResult` (see `contracts/trace-schema.md`) per pair.
- Findings with `generator=context_isolation`, typically class `handoff_carried_by_external_context`.
- A per-run Isolation Summary: how many handoffs pass isolation, how many fail, which are the most load-bearing.

## Behavior

### B1. Run the baseline (full-content mode)

Execute u → v using component 05's normal grounded mode with full seed context. Capture v's observed outputs and outcome. This is the "baseline" — what the system currently does.

### B2. Construct the isolated context

Start from the same seed context, but:
- Strip out any state paths not declared in some upstream triple's `pim.state_writes` or `pim.postconditions`. I.e., only information that the declared contracts guarantee remains.
- Execute u → v again, but when handing context to v, include **only** what u's `pim.state_writes` and `pim.postconditions` declare.
- Include u's full PSM content if configured (test both modes: strict-contract and contract-plus-u-content).

Three isolation levels, run as separate configurations:

- **Level 1 — declared_outputs_only:** v receives only u's declared writes. Strictest test.
- **Level 2 — declared_outputs_plus_u_content:** v additionally sees u's PSM content. Tests whether v uses u's declared interface or scans u's raw content for information.
- **Level 3 — full_context_minus_undeclared_state:** v sees full declared state history but no undeclared fields. Tests whether v relies on fields that leaked into state without declaration.

### B3. Compare outcomes

For each isolation level, compare v's output against baseline:
- **Identical output → clean handoff.** The declared contract was sufficient.
- **Different output → isolation failure.** The handoff relied on something outside the declared contract.

Comparison dimensions:
- `observed_writes` paths match?
- `observed_writes` values match (modulo type-appropriate tolerance)?
- Branch choice (if gateway) matches?
- Refusal status matches?

### B4. Classify divergence

When divergence detected, diagnose:

- **Missing information:** isolated run refuses or produces incomplete output; baseline produces complete output. → Conclusion: u is not declaring all the outputs v needs, OR v is reading state u didn't declare writing. Both are `handoff_carried_by_external_context`.

- **Different choice:** both runs produce output but differ (e.g., different branch, different field values). → Conclusion: v is making its decision based on undeclared context. Strong `handoff_carried_by_external_context`.

- **Silent divergence:** isolated run produces an output that looks reasonable but differs from baseline. This is the most insidious case — the contract-based execution looks fine but behaves differently. Requires SME review to confirm which outcome is "correct."

### B5. Build the divergence signature

For each divergent pair:
- `missing_without_content` — paths baseline wrote that isolated did not.
- `extra_with_content` — paths isolated wrote that baseline did not (rare but possible).
- `behavior_change` — natural-language description generated by comparing traces.

### B6. Aggregate isolation findings

At end of isolation run across many pairs:
- **Ranked load-bearing list:** pairs with the most severe divergence, ranked. These are the handoffs most critically carried by external context.
- **Systemic diagnosis:** if many divergences share a pattern (e.g., most involve a specific state namespace), that's a systemic finding.
- **Clean handoffs count:** pairs with identical outcomes. This is a positive KPI for the corpus.

### B7. Calibration mode

Because this component's false-positive rate directly affects its usefulness, support a calibration mode:
- SME provides a set of handoffs they have manually verified as "should be clean."
- Run isolation against those.
- Any divergence on SME-clean handoffs is either a real finding (SME was wrong) or a false positive (harness needs tuning).
- Track the false-positive rate; if > 20%, the harness needs adjustment before findings are trusted at scale.

## Public API

```python
class ContextIsolationHarness:
    def __init__(self, graph: JourneyGraph, llm: LLMClient, config: IsolationConfig): ...
    def isolate_pair(self, u: str, v: str, seed_context: JourneyContext, level: IsolationLevel = Level.DECLARED_OUTPUTS_ONLY) -> IsolationResult: ...
    def isolate_journey(self, trace: Trace, level: IsolationLevel = Level.DECLARED_OUTPUTS_ONLY) -> list[IsolationResult]: ...
    def calibrate(self, known_clean_handoffs: list[tuple[str, str]], seed_contexts: list[JourneyContext]) -> CalibrationReport: ...
```

## What this component does NOT do

- Does not generate personas.
- Does not fix contracts — reports which handoffs are load-bearing.
- Does not judge which outcome (baseline or isolated) is "correct" — just reports that they differ.
- Does not re-run static checks — complements them.

## Dependencies

- Component 03 (Journey Graph).
- Component 05 (Grounded Handoff Runner) — reused for actual execution in each mode.
- Component 09 (Persona Generator).
- LLM client.
- `contracts/trace-schema.md`, `contracts/finding-schema.md`.
- Component 12 (Finding Store).

## Verification

**V1.** On a pair where u declares its output completely and v consumes only declared fields: baseline == isolated, zero isolation findings.

**V2.** On a pair where u declares writing `field_a` but also (undeclared) leaks `field_b` into state, and v silently uses `field_b`: isolated run diverges from baseline — one `handoff_carried_by_external_context` finding with `missing_without_content: [field_b]`.

**V3.** On a pair where u's PSM content contains information v needs but u doesn't declare it as a state write: Level 1 diverges, Level 2 converges. Diagnosis: v is reading u's content rather than u's declared state. Finding tags this as a specific subtype.

**V4.** Calibration mode: given a SME-marked clean set, tracked false-positive rate is reported and stays bounded.

**V5.** Determinism: re-running isolate_pair with same seed and config yields same IsolationResult.

**V6.** Performance: isolation adds ~2x the grounded runtime cost per pair (two executions instead of one, plus analysis). Budget controls enforced.

## Implementation notes

- This is the single most important component to get right. If it produces false positives at a high rate, SMEs will learn to ignore it, and the simulator loses its most valuable diagnostic.
- Invest in good divergence signatures — the `behavior_change` field should be readable, not cryptic JSON diffs.
- Run the full-content baseline first and cache it; isolated runs compare against cache.
- For Level 1 (strictest), it's likely almost every handoff will diverge initially, because most authoring does not fully declare outputs. That's not a bug in the harness — it's the finding. Expect a large initial finding volume and plan triage accordingly.
- Consider a progressive-disclosure UI for results: top 20 most load-bearing handoffs first, drill down to per-pair detail.
- The calibration step should precede the first production run, not come after. A harness emitting many unclassified findings without calibration baseline will destroy its own credibility.
